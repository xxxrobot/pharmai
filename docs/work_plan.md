## 三、风险评估

### 3.1 迁移过程中的兼容性风险

| 风险 | 可能性 | 影响 | 缓解措施 |
|------|--------|------|----------|
| API 破坏 | 中 | 高 | 保持向后兼容的 wrapper |
| 模型格式变更 | 低 | 高 | 模型版本控制 + 转换脚本 |
| 性能回退 | 中 | 中 | 基准测试对比 |
| 依赖冲突 | 低 | 中 | 虚拟环境隔离 |

**具体建议:**
1. **分阶段迁移**: 先并行运行新旧代码，验证一致性后再切换
2. **兼容性层**: 保留根目录脚本的 facade 模式，内部调用 core 模块
3. **回归测试**: 每次迁移后进行完整的端到端测试
4. **版本锁定**: 锁定关键依赖版本 (rdkit, scikit-learn)

### 3.3 依赖管理建议

**当前依赖状态:**
```
主要依赖:
├── rdkit>=2023.0.0      (化学信息学核心)
├── scikit-learn>=1.3.0  (机器学习)
├── pandas>=1.5.0        (数据处理)

├── streamlit>=1.28.0    (Web界面)
├── deepchem>=2.7.0      (深度学习)

```

**建议改进:**
1. **依赖锁定**: 创建 `requirements-lock.txt` 锁定确切版本
2. **开发依赖分离**: 创建 `requirements-dev.txt` 包含测试工具
3. **可选依赖**: 将 deepchem 设为可选 (体积大，非必需)
4. **安全扫描**: 引入 `safety` 或 `pip-audit` 定期检查漏洞

**推荐的 requirements 结构:**
```
requirements.txt          # 核心依赖 (最小集合)
requirements-dev.txt      # 开发依赖 (测试、lint)
```
